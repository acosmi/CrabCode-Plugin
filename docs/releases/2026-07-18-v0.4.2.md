# crabcode-media-ops 0.4.2 — 分发验证硬化版

> 日期：2026-07-18
>
> 依据：`docs/audit/2026-07-18-crabcode-media-ops-MCP不可用根因审计与实施方案.md` 交付后暴露的 CI 侧残留缺口（分发门禁从未在 Linux 实跑）。

## 背景：0.4.1 交付时标注的残留缺口

0.4.1 把自包含 `dist/server.js` 与 `check:distribution`（免 git 新鲜度比对 + 清洁目录冷启动 smoke）接入 CI，但该产物在 **Windows** 上 `bun build` 并入库，其 **Linux 字节确定性未经 CI 验证**——因为 media-ops 的 CI job 运行在 Playwright 官方容器镜像内，而该镜像**不含 `unzip`**，`oven-sh/setup-bun@v2` 无法解包 bun 二进制，导致 `setup-bun` 直接失败、其后 `typecheck / check:distribution / test / test:qa / qa:release / validate` **全部被跳过，从未在 Linux 运行过**。

## 变更摘要

| 类别 | 变更 |
| --- | --- |
| CI 修复 | `.github/workflows/ci.yml` 的 `media-ops-plugin` job 在 `setup-bun` 前补装 `unzip`（容器内 root，`apt-get install -y --no-install-recommends unzip`）；`verify` 与 `html-video-plugin` 运行于 `ubuntu-latest`（自带 unzip）不受影响、不改动 |
| CI 可靠性 | `playwright.config.ts` 测试超时由默认 30s 提升至 120s——`delivery-qa` 全 QA 浏览器用例（Nu+Chromium+axe 全链）运行时间随跑机负载波动（本地容器多次实测 14.6s / 23.4s / 30.4s），曾达 30.4s 触顶 30s 默认上限而纯超时失败（非断言失败，同码在 `test:qa` 的 180s 预算下稳过）；提升至 120s 留足余量，属该用例首次在 Linux 实跑后暴露的配置校正，不入包、不影响产物 |
| 产物权威性 | `dist/server.js` 以**精确 CI 容器（digest 锁定）内 Linux `bun build`** 权威重建入库；`docs/legal/SBOM.cdx.json` 同步重生成至 0.4.2 |
| 版本 | `plugin.json` / `package.json` / `marketplace.json` 条目 / `src/domain.ts` `VERSION` / `README.md` 标题 / SBOM 组件版本一致提升至 0.4.2 |

## 根因与验证（本机精确复现，非只看表面）

- **根因确证**：拉取 CI 所用镜像（按 digest `sha256:5b8f294a…` 锁定）在容器内实测 `command -v unzip` 为空——镜像确实缺 `unzip`，`setup-bun` 因此失败。这是 CI 长红且 media-ops job 从未跑过真实步骤的唯一原因。
- **全 job 转绿**：在“Playwright 官方镜像 + 从 Docker Hub 取的 Linux bun 1.3.11 + `unzip`/JRE21”派生镜像内，克隆入库 HEAD、经国内镜像 `bun install --frozen-lockfile`（199 包）后依次实跑，全部通过：`check:distribution`（`status: passed`，38 工具，冷启动 smoke 467ms < 10s 预算；历次数百 ms–数 s，恒稳过）、`typecheck`、`test`（109 pass / 0 fail）、`test:qa`（Nu/axe/Playwright 全链，1 pass）、`qa:release`（8 项浏览器用例含 `maxDiffPixels:0` 像素级黄金截图 + “连续 100 次字节一致”）、`validate`（SBOM 200 组件对齐）。
- **dist Linux 字节确定性**：`check:distribution` 在 Linux 重建 `server.js` 与入库产物归一化后字节一致——先前 Windows 构建产物在 Linux 亦可复现，缺口关闭。本版进一步以 Linux 容器构建为权威入库。
- **Windows 本机既有测试失败为环境噪声**：0.4.1 标注的 Windows 上 package/storage/approval 文件锁（EBUSY/EPERM）失败，在 Linux 容器内 109 单测全过，确认与产物无关。

## 兼容与回滚

- 运行时**功能与 0.4.1 完全等价**：本版仅 `VERSION` 串更新与 `dist` 平台权威重建，无工具增删（仍 38）、无 schema 变更（`SCHEMA_VERSION=2` 不变）、无存储迁移。
- 已安装 0.4.1 缓存不被修改；升级经 marketplace 正常分发。回退到 0.4.1 无副作用（产物行为一致）。

## DoD 门禁（全部在精确 CI 容器内 Linux 通过）

```bash
cd plugins/crabcode-media-ops
bun run typecheck            # 双配置全绿
bun run check:distribution   # 新鲜度 + 冷启动 smoke:38 工具，1282ms < 10s
bun run test                 # 109 pass / 0 fail
bun run test:qa              # 全自动交付 QA:1 pass
bun run qa:release           # 8 项浏览器用例（含像素级黄金截图）全过
bun run validate             # sbom:check + 契约:aligned at 0.4.2
```

## 残余风险与后续

- 本版只修复并验证 media-ops job。仓库 `verify` job 仍因既有的根测试失败（CrabAccount Bash client、workflow-skill-presentation-completeness）而红——属独立的预存问题，不在本次范围。
- `check:distribution` 现已在 Linux CI 实跑，后续任何 `dist` 变更都会被 CI 的跨平台字节门禁捕获，无需再依赖本地容器手工复现。
