### C. 下游「管家工作流」模型关键摘录(出自 HTML v1.0,2026-06-20)

- **四个核心概念**:
  - **工作流(Workflow)**:围绕职能预设的能力集合,作为顶部"插槽"常驻。预设 4 个——**法律、媒体运营、文档表格、财务分析**,可自定义新增。
  - **插槽(Slot)**:工作流在顶部栏常驻位,跨会话不消失,支持热插拔,右侧靠齐。
  - **能力卡片(Capability)**:每个工作流下固定挂载的**插件 / 技能 / 命令**;选中工作流后在输入框下方**网格展示**,点击带入输入框。
  - **会话(Session)**:按工作流聚合归档到左侧菜单分组。
- **生命周期 5 阶段**:① 初始化空态(插槽常驻 + 3 引导卡片)② 挑选工作流(三区联动:顶部芯片高亮 / 中部换成能力卡片网格 / 左侧分组高亮;**意图自动推荐**已落地)③ 调用能力(点卡片注入提示词模板 / 键入 `/` 或 `@` 内联唤起插 token 芯片)④ 发送执行(插槽始终常驻,不随发送消失)⑤ 归档复用(入左侧工作流分组)。**贯穿全程:热插拔**(随时挂载 / 二次确认卸载,历史会话始终保留)。
- **能力卡片布局约束**:卡片**一排 4 个**、略超输入框宽度,**单工作流一行容纳,多选时尽量一屏不滚动**。→ 隐含单工作流能力数适中(约 4–12),这是法律 105 / 财务 54 必须**分组**的根据。
- **分组渲染已落地(支撑 A2 的关键)**:优化建议③——"多工作流并用时,顶部芯片可多选同时高亮,**中部能力区按工作流分组合并展示**,带「已并用 N 个工作流」头部与「清空选择」"。A2 即把此分组从"跨工作流"复用到"工作流内按板块"。
- **能力 token 配色规范**:技能 / 插件 = 橙;命令 = 蓝;插件徽章 = 紫。**无 agent 类型**(故 agents 不进卡片层,仅作内部实现)。
- **模板可填槽位已落地**:带入提示词的关键变量为虚线高亮槽,点击循环切换备选值。
- **能力库入口已落地**:工具栏「能力库」按钮或输入 `/skills` 打开双栏菜单(左列工作流、右列能力)。
- **中优先未落地**:⑤ 自定义工作流的信任治理(展示携带插件/权限/来源/版本,先审后挂载)、⑥ 会话与工作流列表统一视觉与稳定排序。

## 〇、下游模型(对标对象)与仓库现状(被审计对象)

**下游「管家」是严格两层模型**:
- **工作流(Workflow)** = 顶部常驻插槽,可热插拔。预设 4 个:法律、媒体运营、文档表格、财务分析,且可自定义新增。
- **能力卡片(Capability)** = 工作流下固定挂载的 **插件 / 技能 / 命令**,选中工作流后在输入框下方**网格展示**(一排 4 个,单工作流尽量一行/一屏不滚动),点击带入输入框。

**它对仓库的隐含要求**:
1. 每个"工作流"必须是**一个可独立安装的主插件**(下游"装一个主插件工作流")。
2. 该插件须能被**机械枚举出它的能力卡片清单**(skills / commands)。
3. 市场需能区分"哪些条目是工作流(上插槽)/哪些是能力或连接器(进卡片或挂载)"。
4. 单工作流能力数宜适中(卡片网格"一行/一屏"隐含约 4–12)。

**仓库现状核实(89 条目 / 70 目录 / 15 category)**:

| 物理深度 | 代表 | 数量 | 是否对齐两层模型 |
|---|---|---|---|
| **1 层 · 原子** | 12 LSP、11 connector、output-style×2、frontend-design/math-olympiad/playground/session-report/skill-creator 等单组件 | ~30 | ❌ 插件本身就是一个能力,无"工作流"外壳 |
| **2 层 · 伞插件** | crabcode-office-suite、crabcode-media-ops、14× crabwork-* | 16 | ✅ 伞插件(skills[]) → 能力,正是模型期望 |
| **3 层 · 容器** | crablaw-cn(13 板块/105 skill)、crabfin-cn(7 板块/54 skill) | 2 容器 / 20 条目 | ❌ 容器目录 → 板块伞插件 → skill,超出两层 |

**下游 4 预设工作流的落地核实**:

| 预设工作流 | 仓库对应 | 结构 | 判定 |
|---|---|---|---|
| 文档表格 | `crabcode-office-suite` | 2 层,skills[]=4 | ✅ 干净,可作样板 |
| 媒体运营 | `crabcode-media-ops` | 3 skill + 6 cmd + 5 agent,**不声明 skills[]** | ⚠️ 能力靠目录扫描;agent 无对应卡片类型 |
| 法律 | `crablaw-cn` | **纯容器**(无顶层 plugin.json),13 个并列 marketplace 条目,105 skill | ❌ 装不到"一个主插件" |
| 财务分析 | `crabfin-cn` | **纯容器**(无顶层 plugin.json),7 个并列条目,54 skill | ❌ 同上 |

> **一句话**:4 个旗舰预设里,只有 1 个(文档表格)是模型期望的干净结构;其余 3 个分别"组件未声明""容器拆成 13 条""容器拆成 7 条"。

## 一、第一阶段:根因审计

### 根因 ① — marketplace 缺「类型 / 层级」分层法,89 条扁平异构表无法机械映射为两层模型

- **问题描述**: 下游要把市场条目渲染成 工作流(插槽) + 能力卡片(网格) 两层;但 marketplace.json 是同一层的 89 条扁平 list,每条仅含 `category`/`tags`,**无字段标明该条目在产品模型里扮演的角色**(工作流 / 能力卡片 / 连接器)。
- **根因(非症状)**: 缺少**条目类型(type)与层级(tier)分类法**。`category` 只表达领域(法律/财务/语言服务…),不表达"在管家模型里是什么"。
- **影响面 / 关联方**: 下游前端无法自动决定谁上插槽、谁进卡片;产品侧只能硬编码 4 预设并手工维护"工作流→能力"映射;插件作者新增条目无法自动归位。受影响:下游前端、产品运营、所有插件作者、未来第三方/自定义工作流。
- **是否症状级**: 是补丁则在此复发——若仅手工维护一张"预设→能力"映射表,每次新增/重构插件都要改表,且**下游已规划的"自定义工作流热插拔挂载"**(文档阶段 3)处第三方插件无法自描述其角色,必然失效。
- **同类变体**: 全部 89 条目同病;自定义/第三方工作流场景完全无解。

### 根因 ② — 同一个「工作流」概念在仓库里 1/2/3 层三种物理深度并存

- **问题描述**: 下游是严格两层;仓库却同时存在 1 层原子(~30)、2 层伞(16)、3 层容器(2)。其中容器目录 `crablaw-cn`/`crabfin-cn` **自身没有顶层 plugin.json,根本不是可安装条目**,直接违背"装一个主插件工作流"。
- **根因(非症状)**: 缺统一的「工作流粒度」定义 + 缺「一个工作流 = 一个可安装的伞插件条目」的硬约束。容器目录被当作"分组文件夹"用,而下游模型里没有这一层。
- **影响面 / 关联方**: 下游 4 预设的 2 个旗舰(法律/财务)落不到单个可装插件——用户想"装法律工作流",实际要在市场里逐个挑装 13 个并列条目。受影响:法律/财务全体用户、下游插槽挂载逻辑。
- **是否症状级**: **06-20 审计已识别 Model A(伞)vs Model B(容器)矛盾并建议重构,但此后问题不是被修复而是被放大**——当时 crablaw-cn 仅 4 板块、且 crabfin-cn 尚不存在;中国法化把 crablaw-cn 撑到 13 板块/105 skill,又**按同一容器模式新建了 crabfin-cn** 7 板块/54 skill。症状级处理(只归并 category、不动结构)正是它复发并蔓延到财务的原因。
- **同类变体**: crabfin-cn 是 crablaw-cn 的同模式复制;未来任何"大域中国法化/本地化"都会再触发此容器模式。

### 根因 ③ — 大域工作流的能力规模远超下游卡片网格容量(法律 105 / 财务 54 / small-business 31)

- **问题描述**: 下游卡片"一排 4 个、单工作流尽量一行/一屏不滚动",隐含单工作流约 4–12 能力。若把 法律 当单工作流则 105 张卡(≈26 行);财务 54 张;此外 `crabwork-small-business` **单个伞插件就含 31 skill**。
- **根因(非症状)**: 下游模型设计时**默认每工作流能力数适中**,未覆盖"中国法化后单域能力爆炸"的真实数据;仓库侧也缺"卡片=精选入口" vs "全量 skill=渐进披露"的区分——所有 skill 一律平铺。
- **影响面 / 关联方**: 任何"大域即单工作流"的方案都会撑爆卡片区,迫使下游加滚动(体验退化)或加分组(UI 改造)。受影响:下游交互、所有重度域。
- **是否症状级**: 靠 UI 滚动硬塞是症状级,能力越加越糟。根因解需要"卡片层 ≠ 全量能力层"的分层。
- **同类变体**: small-business(31)已在 crabwork 域内先行暴露同样的"单插件能力过密"。

### 根因 ④ — 2 层伞插件内部的能力声明不统一,多组件插件不声明能力清单

- **问题描述**: `crabcode-media-ops`(3 skill+6 cmd+5 agent)、`code-modernization`、`pr-review-toolkit`、`plugin-dev`、`feature-dev` 等约 15 个多组件插件**不声明 skills[]**,能力只能靠目录扫描;且 **agents 这一组件在下游卡片配色规范里没有对应 token 类型**(下游只有 技能/插件=橙、命令=蓝、插件徽章=紫)。
- **根因(非症状)**: 缺"工作流条目必须显式声明其能力卡片清单(skills + commands)"的统一约束;agents 作为内部实现未纳入卡片模型。
- **影响面 / 关联方**: 这些插件的能力无法被下游稳定枚举成卡片;agents 无处展示。受影响:下游枚举逻辑、上述 15 个插件。
- **是否症状级**: 逐个补 skills[] 治标;根因是缺统一 manifest 约束(应在转化规范里固化)。
- **同类变体**: 全部 multi-no-decl 插件(约 15 个)。

### 三个总问(强制回答)

1. **是否架构偏差**:**是**。marketplace 缺类型/层级分层(根因①)+ 工作流三种物理深度并存(根因②)= 与下游两层模型的结构性偏差,非表层瑕疵。
2. **是否过度工程**:**局部有苗头**。crablaw-cn 13 板块/105 skill、crabfin-cn 54 skill、small-business 31 skill,相对下游卡片容量属"过度细分 / 过度平铺"。但**内容本身有价值,问题在"未分层暴露"而非"该删"**——根因解是分层,不是裁内容。
3. **是否遗漏同类 / 低估关联方**:**06-20 审计低估了**。它把 crablaw-cn 当 4 板块的局部问题,既没预见中国法化会放大 3 倍,也没预见会复制到财务;更没把 ~30 个 1 层原子插件纳入下游两层模型(它们在模型里目前**无位置**)。本审计补齐这两块。

## 二、第二阶段:实施方案(决策门 — 详细步骤待用户拍板后定稿)

整体方向(四条根因对应):
- 对①:给 marketplace 条目引入**类型/层级元数据**(如 `tier: workflow | capability | connector`、`group: 法律 | 财务 | …`),让下游可机械映射,无需硬编码。
- 对②:确立硬约束「**一个工作流 = 一个可安装的伞插件条目**」,消除"纯容器目录"这一既非工作流又非能力的中间态。
- 对③:区分「能力卡片(精选入口)」与「全量 skill(渐进披露)」,大域不把全部 skill 平铺成卡。
- 对④:在转化/作者规范里固化「工作流条目必须显式声明能力卡片清单」。

**用户已拍板架构取向**(2026-06-21):
- **决策 A = A2**:大域即单工作流,板块降为分组。crablaw-cn / crabfin-cn 各收敛为**一个可安装的伞插件工作流**,13 / 7 板块作为其能力的**分组(group)**;且**插件市场里分组也要显示**,以兼顾"装整个法律工作流"与"按板块浏览/选用"两种使用场景。法律、财务统一按此逻辑。
- **决策 B = B1**:marketplace 条目加 `tier` 字段区分 **工作流(workflow) / 连接器(connector) / 能力(capability)**。

> A2 与下游 HTML 的一致性论证:下游优化建议③已落地"**多工作流并用时,中部能力区按工作流分组合并展示**(带「已并用 N 个」头部)"。A2 的"工作流内按板块分组"是把这套**既有分组渲染**从"跨工作流"复用到"工作流内",数据模型同构,不是全新 UI 能力。

### 0 · 目标数据模型

```
marketplace 条目 = {
   ...现有字段,
   tier: "workflow" | "connector" | "capability",      // B1 新增
   groups?: [ { name, displayName, skills:[skillName...] } ]   // A2:仅 workflow 且有板块时
}
```
- **tier=workflow**:下游可上顶部插槽、展开能力卡片网格。
- **tier=connector**:MCP / LSP 桥接,不占插槽,作为可挂载连接器或工作流下的依赖。
- **tier=capability**:单一能力 / 样式 / 示例,既非职能工作流也非连接器。
- **groups**:有板块的工作流(法律/财务)用它驱动"卡片按板块分组"与"市场内分组显示";无板块工作流(office-suite、多数 crabwork-*)省略,卡片平铺即可。

### 1 · 全条目 tier 分类(规则 + 归属)

判定规则(从严,边界从保守):
- **workflow**:职能工作流伞 / ≥3 个能力组件(skills+commands)的工具箱。
- **connector**:主要资产是 `.mcp.json`(MCP)或 LSP 桥接,自有 skill/command ≤ 薄封装。
- **capability**:仅 1 个 skill/command/agent、或 output-style / 示例 / 安装器。

### 5 · 根因判定硬标准(反证)

- **根因①(无类型分层)**:加 `tier` 后,下游可机械区分插槽/卡片/连接器,**新增任意插件只要带 tier 即自动归位**——若不解此根因,则每加一个插件都要改下游硬编码映射,在"自定义工作流热插拔"处复发。tier 字段使该复发点消失。
- **根因②(三层物理深度)**:法律/财务收敛为单伞条目后,"纯容器目录"中间态被消除,**"装一个主插件工作流"成立**;反证——若仍保留 13/7 并列条目,用户装"法律"仍需逐个挑装,问题原样复发。
- **根因③(卡片爆炸)**:`groups` 把 105/54 能力按板块分组渲染(复用下游既有分组),不再平铺成 26 行;反证——若不分组,卡片区无论怎么调都会溢出一屏。
- **根因④(声明不统一)**:补 skills[] 后能力可机械枚举;反证——不补则 media-ops 类插件能力对下游不可见。
