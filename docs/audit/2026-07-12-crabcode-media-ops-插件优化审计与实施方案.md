# CrabCode 媒体运营插件优化审计与实施方案

> 日期：2026-07-12
>
> 审计对象：`plugins/crabcode-media-ops/`
>
> 对照对象：`plugins/crablaw-cn/`、`plugins/crabcopyright-cn/`、`wechat-original-opinion.skill`
>
> 当前阶段：方案已于 2026-07-12 全量实施；复核结果见同目录实施复核审计报告
> 方法：按 skill-creator 的“现状基线 → 测试 → 改进 → 对比复核”方法规划后续工作

## 一、结论摘要

不建议把 `wechat-original-opinion.skill` 原样复制进 `crabcode-media-ops`。

建议把媒体插件升级为一个“小型伞插件”：借鉴法律插件的共享底座、专业板块、状态契约、角色隔离和机器门禁，同时保留上传技能中成熟的公众号原创观点、证据链、原创复核和风格学习能力。

实施顺序应当是：

1. 先修复审批和事实核查门禁可以被绕过的问题；
2. 再建立媒体插件共享底座和内容状态契约；
3. 然后模块化吸收上传技能；
4. 最后升级平台规则、中文热点算法、评测与发布文档。

### 1.1 补充强制实施决策（R-01）

**创作者风格采集表单是本次媒体插件优化的强制交付能力，不是可选建议、探索项或后续 Gate B 能力。**

本次优化必须完成：

- 独立的 `media-style-intake` 技能；
- 快速、完整、增量三种表单模式；
- 表单草稿保存、恢复、提交、确认和废止状态；
- 创作者类型、内容类型、表达风格、结构偏好、真实性边界、样本权利和平台差异字段；
- 表单结果与历史作品分析结果的冲突比对；
- 未经创作者确认不得覆盖正式 profile；
- 按 `brand_id` 隔离的表单、profile 版本、历史和回滚；
- 对应 schema、确定性工具、单元测试、技能测试和 README 使用说明。

该能力未完成或只用一段对话提示临时替代时，本次插件优化不得判定为完成，也不得进入正式版本发布。

本次只读审计期间：

- 根仓库 `bun run validate` 通过；
- 媒体插件 TypeScript 类型检查通过；
- 媒体插件现有测试 `49/49` 通过；
- 工作区在审计结束时无实现改动。

现有测试通过不代表工作流门禁完整。审批绕过等问题属于“测试未覆盖的设计缺口”，不是现有测试失败。

## 二、现状盘点

### 2.1 媒体插件现有组成

排除 `node_modules` 后，当前媒体插件主要包含：

- 3 个技能：`media-ops`、`media-human-editor`、`media-platform-adapter`；
- 7 个命令；
- 7 个子代理；
- 5 份共享参考资料；
- 17 个 TypeScript 源码文件；
- 7 个测试文件；
- 一个 Gate A 本地 MCP 服务。

现有设计的正确方向包括：

- 创作由 agent 完成，MCP 只做确定性操作；
- 选题、brief、主稿、审稿、平台变体、预览和发布包形成基本闭环；
- 品牌 profile 与稿件关联；
- 事实核查采用 `verified / doubtful / unsourced` 三态；
- 存疑项要求修订或具名人工放行；
- 真平台发布与评论发送明确留在 Gate B；
- 内容法律风险可以路由至 `crablaw-cn`。

### 2.2 上传技能的组成

`wechat-original-opinion.skill` 包含：

- 一个微信公众号原创观点主技能；
- 联网可信源核验标准；
- 微信公众号编辑手册；
- 原创与发布风险复核标准；
- 微信平台规则核验资料；
- 风格档案、风格学习规范和变更日志；
- 本地短语重合与编辑风险扫描脚本；
- 风格样本元数据登记脚本。

其最有价值的能力是：

- 先建立主张台账，再进入写作；
- 区分事实、来源解释和作者推论；
- 要求独立论点、因果链、强反方和边界条件；
- 禁止虚构亲测、访谈、读者反馈和第一人称经历；
- 不承诺“原创率”或微信原创声明结果；
- 风格学习记录来源、权利状态、置信度和不可复制表达；
- 原创扫描明确承认其只是用户提供文本范围内的启发式预警。

## 三、关键问题与优先级

### P0-1：人工审批门禁可以被绕过

现状：

- `mediaops.approval.request` 只能创建 `pending` 记录；
- 没有批准、拒绝、撤销或失效动作；
- `mediaops.publish.package` 的 `approvalId` 是可选字段；
- 发布包工具不查询审批记录，也不检查审批状态；
- 即使没有审批记录，仍可直接生成发布包。

影响：

文档宣称的“人工审批硬 gate”只存在于提示词层，没有在确定性工具层兑现。

根因：

审批被设计成一条附加记录，而不是内容状态机中的强制状态迁移。

修复原则：

- 新增审批决定和查询能力；
- 审批必须绑定内容 ID、目标平台、内容哈希、批准人和时间；
- 发布包必须读取一条有效的 `approved` 记录；
- 当前稿件哈希与获批稿件哈希不一致时，审批自动失效；
- 不允许调用方仅传入一个无法验证的任意 `approvalId`。

### P0-2：事实核查门禁可以被绕过

现状：

`readiness.inspect` 在未传 `claims` 时使用空数组，因此无法区分：

- 已完成核查且没有可验证主张；
- 根本没有进行事实核查；
- 调用方遗漏了审稿结果。

同时，发布包工具不会重新执行完整 readiness 检查。

影响：

未核查稿件可能被当作“没有存疑项”进入发布包。

修复原则：

- 引入明确的 `reviewStatus`；
- 进入发布阶段时必须存在审稿记录；
- `claims: []` 只有在审稿记录明确说明“未发现可验证主张”时才有效；
- 发布包工具进行 defense-in-depth 复核；
- 存疑放行记录绑定具体主张和具体稿件版本。

### P0-3：AI 标识法律口径写得过满

现状：

插件把固定正文文案“本文由 AI 辅助创作”描述成所有发布场景唯一的法律硬要求，并通过正文子字符串检查强制执行。

官方规则核对：

- 《人工智能生成合成内容标识办法》第十条要求用户发布生成合成内容时主动声明，并使用传播服务提供者提供的标识功能；
- 规则区分显式标识、隐式标识、生成服务提供者、传播服务提供者和发布用户；
- 规则没有规定所有平台正文必须使用唯一固定句子。

官方来源：

- https://www.cac.gov.cn/2025-03/14/c_1743654684782215.htm

修复原则：

- 保留严格合规策略，但不把插件固定文案表述成唯一法定形式；
- 数据模型区分 `platform-native`、`body-label`、`file-metadata` 等披露方式；
- 平台原生声明功能优先；
- 正文标识作为平台能力不足、发布包手动流转或内部策略需要时的安全兜底；
- 发布包记录实际采用的披露方式、规则版本和人工确认情况。

### P1-1：新旧技能触发冲突

现状：

- 当前 `media-ops` 用“写一篇公众号”作为触发示例；
- 上传的 `wechat-original-opinion` 同样要求“写篇有观点的公众号文章”触发；
- 上传技能还同时承担研究创作、重写、原创复核、采风学习和风格更新五种模式。

影响：

同一个请求可能同时匹配总编排技能和公众号专业技能，造成入口不稳定、重复工作或错误路由。

修复原则：

- `media-ops` 只负责多阶段、多平台或走到审批发布包的完整运营流程；
- `wechat-original-opinion` 只负责微信公众号原创观点、深度评论和重写增深；
- 原创复核与风格管理拆成独立技能；
- 用 near-miss 触发测试验证公众号观点文、品牌软文、多平台营销稿、纯新闻摘要等边界。

### P1-2：上传技能的风格存储方式不适合插件化

现状：

- `style_memory.py` 默认写入技能安装目录中的 `references/style-sources.json`；
- 工作流要求直接修改 `style-profile.md` 与 `style-changelog.md`；
- 默认风格固定为“阿拙式 AI 外骨骼评论”。

影响：

- 插件安装目录可能只读；
- 插件升级会覆盖用户学习结果；
- 多个品牌共用一个全局风格档案会串味；
- 默认人格可能污染并不适合该风格的账号。

修复原则：

- 沿用 `${CRABCODE_PLUGIN_DATA}/profiles/<brand_id>/` 品牌隔离；
- 增加独立的“创作者风格采集表单”，让没有历史文章、历史样本不足或希望主动定义风格的创作者也能完成首次建档；
- 表单先生成 profile 草案，不得未经创作者确认直接覆盖正式 profile；
- 风格档案采用版本化、append-only 历史；
- 风格更新先生成 diff，经人工确认后写入；
- 支持回滚、来源元数据和权利状态；
- “阿拙”只作为可选 preset，不作为整个插件默认人格；
- 不保存整篇外部文章，只保存必要元数据和抽象特征。

创作者风格采集不能只依赖“让模型看几篇文章后猜”。目标风格、真实身份边界、内容类型、读者对象、语言禁区和商业合规要求，应允许创作者通过结构化表单主动填写；历史作品分析作为证据补充，与表单结果发生冲突时必须展示差异并交由创作者确认。

### P1-3：内容状态没有统一契约

现状：

`content.save` 接受 `z.record(z.unknown())`，brief、draft、variant 的实际结构没有代码级约束。

影响：

- 无法可靠校验父稿、来源、品牌版本、审稿结论和审批之间的关系；
- “更新稿件”实际上可能只是继续追加一条新记录；
- 无法判断审批针对的是哪个具体版本；
- 运行记录难以跨技能可靠交接。

修复原则：

建立统一 `content-manifest`，作为跨技能单一事实源。

建议核心字段：

```json
{
  "contentId": "...",
  "brandId": "...",
  "profileVersion": "...",
  "stage": "intake|researched|drafted|reviewed|approved|packaged",
  "parentIds": [],
  "claims": [],
  "citations": [],
  "originalityReview": {},
  "legalReview": {},
  "platformRuleVersion": "...",
  "aiDisclosure": {},
  "approval": {},
  "artifacts": [],
  "auditEvents": []
}
```

### P1-4：中文热点聚类基本失效

现状：

热点聚类先按空格分词。中文标题通常没有空格，因此整条中文标题会成为一个 token，Jaccard 相似度只有在标题几乎完全一致时才可能达到阈值。

修复原则：

- 中文采用字符 bigram/trigram 或可靠的轻量分词方案；
- 中英混合标题分别处理；
- 增加同事件不同表述、旧闻复炒、主体相同事件不同等中文测试；
- 聚类结果继续保持可解释，不引入不可审计的黑箱分类作为唯一判断。

### P1-5：平台规则缺少来源、核验日期和失效策略

现状：

微信公众号、小红书、今日头条的标题、正文和图片限制直接硬编码在 `registry.ts`，没有逐条记录：

- 官方来源；
- 核验日期；
- 适用内容类型；
- 是官方硬限制还是运营建议；
- 规则过期后的降级行为。

修复原则：

- 建立版本化平台规则包；
- 每条规则带 `sourceUrl`、`verifiedAt`、`scope`、`ruleType`；
- 区分硬限制、平台动态规则和编辑建议；
- 过期时提示重新核验，不用旧记忆补写“最新规则”；
- 发布当天仍保留人工后台复查项。

微信原创声明是平台比对功能，不是公开的通用原创率分数：

- https://ipr.tencent.com/policy/content/3

公众账号运营者应建立选题、编辑、发布、推广和互动评论的全过程审核机制：

- https://www.cac.gov.cn/2021-01/22/c_1612887880656609.htm

### P1-6：版本口径漂移

现状：

- 插件 manifest：`0.2.0`；
- marketplace：`0.2.0`；
- MCP package：`0.1.0`；
- MCP server 与 capabilities 返回：`0.1.0`；
- capabilities 仍描述为 `skeleton + creation loop`。

修复原则：

建立单一版本源或 CI 一致性校验，在下一次实质升级中统一为同一版本。

### P2：测试只覆盖实现，不覆盖技能质量与路由

现有 49 项测试覆盖了部分代码行为，但缺少：

- 审批状态机和审批后改稿失效；
- 未核查稿件阻断；
- 发布包对完整 readiness 的二次校验；
- 中文热点聚类；
- 平台规则过期；
- 多品牌风格隔离与回滚；
- 原创扫描脚本；
- 技能触发边界；
- 新旧技能的输出质量基线对比。

## 四、从法律插件借鉴什么

不建议机械复制 `crablaw-cn` 的体量。媒体插件应复制的是组织原则，而不是技能数量。

### 4.1 共享实践底座

法律插件通过 `matter-core/PRACTICE.md` 集中定义共享 guardrails，各实体技能引用同一底座，并由仓库校验器检查引用关系。

媒体插件应建立 `media-core/PRACTICE.md`，集中定义：

- 内容输入视为不可信数据；
- 来源等级与引用规则；
- 事实、来源解释、作者推论分层；
- 第一人称真实性；
- 负面评价、隐私和商业合作披露；
- AI 披露；
- 法律插件路由；
- 审稿、审批和发布边界；
- 平台规则时效性。

### 4.2 专业技能分板块

法律插件把合同、诉讼、知识产权等领域拆成边界明确的技能。媒体插件也应将：

- 总编排；
- 研究与选题；
- 公众号原创观点；
- 原创复核；
- 品牌风格；
- 平台适配；
- 发布审批；

拆成独立能力，而不是把所有模式堆在一个技能中。

### 4.3 单一状态源与结构化交接

参考软著插件的 manifest 设计，媒体插件的工序交接不应依赖主会话口头复述，而应读取同一份结构化状态。

### 4.4 对抗式复核与角色隔离

沿用当前媒体插件中 writer 与 fact-checker 分离的正确方向：

- 研究者负责取材和来源；
- 写作者负责论证与表达；
- 核查者只核查，不替写作者掩盖问题；
- 发布者只检查状态和生成包；
- 人工批准权不交给生成者。

### 4.5 CI 强制共享门禁

新增媒体专用校验器，检查实体技能是否：

- 引用 `media-core/PRACTICE.md`；
- 包含 Media Gate；
- 声明输出和下一步；
- 没有死链技能或 MCP 工具引用；
- 不把编辑建议误写成法律或平台硬规则。

## 六、技能边界建议

### `media-ops`

只在以下场景触发：

- 需要选题到发布包的完整闭环；
- 多阶段继续执行；
- 多平台运营；
- 批量内容运营；
- 需要审批、留痕或发布包。

不再用单独的“写一篇公众号文章”作为充分触发条件。

### `wechat-original-opinion`

负责：

- 微信公众号原创观点文章；
- 深度评论；
- 科技、AI、互联网产品和数字生活时评；
- 对现有观点稿进行重写增深。

不负责：

- 多平台营销稿；
- 品牌软文；
- 纯新闻摘要；
- 独立原创风险审计；
- 持久化风格更新；
- 发布审批和发布包。

### `media-originality-review`

只审核，不默认改稿。输出：

- 独立论点；
- 因果链；
- 反方与边界；
- 来源覆盖；
- 近似表达；
- 第一人称真实性；
- 权利、声誉和时效风险；
- `可发布 / 修改后发布 / 需补证` 编辑结论。

结论不代表微信平台审核结果或法律意见。

### `media-style-manager`

负责：

- 风格样本分析；
- 合并创作者风格采集表单与历史作品分析结果；
- 品牌 profile 草案；
- 逐字段 diff；
- 来源和权利状态登记；
- 人工确认；
- 版本、回滚和多账号隔离。

### `media-style-intake`

负责面向创作者提供结构化风格采集表单，既支持首次建档，也支持已有 profile 的定向更新。

表单至少提供三种模式：

- `快速建档`：只收集账号定位、内容类型、目标读者、基本语气、禁用项和代表样本；
- `完整建档`：收集完整风格指纹、真实性边界、平台差异和商业合规要求；
- `增量更新`：只填写希望改变的项目，输出与现有 profile 的逐字段 diff。

该技能只生成待确认草案，不直接写入正式 profile。提交后由 `media-style-manager` 合并历史作品分析、标注冲突与置信度，再经创作者确认后保存。

### `media-publish-gate`

只负责：

- 读取最终变体；
- 核验审稿、原创复核和法律路由状态；
- 核验平台规则版本和 AI 披露方案；
- 请求并记录人工审批；
- 获批后生成发布包；
- 审批后内容变化时阻断。

## 七、创作者风格采集表单设计

### 7.1 表单目标

表单不是简单收集几个“专业、幽默、犀利”标签，而是把创作者希望如何表达、能够以什么身份表达、不能虚构什么，以及不同平台如何变化，转换为可执行且可验证的品牌 profile。

表单必须同时支持：

- 对话式逐步填写；
- 可保存和再次编辑的结构化 JSON 数据；
- 面向非技术用户的可视表单或本地表单预览；
- 从已有 profile 预填后进行增量修改；
- 未完成时保存草稿，完成后进入人工确认。

### 7.2 创作者类型与内容类型

创作者可以多选并排序主要类型，不能被迫只归入一个固定标签。首版必须预置以下类型，并允许自定义扩展：

- 观点评论型；
- 知识科普型；
- 新闻/政策解读型；
- 教程方法型；
- 产品测评/体验型；
- 调查监督型；
- 行业观察型；
- 品牌主理人型；
- 生活方式/个人成长型；
- 故事叙事型；
- 访谈整理型；
- 自定义类型。

每种类型记录：是否为主类型、适用栏目、目标平台、目标读者、期望深度及不适用场景。类型只用于路由和风格提示，不据此虚构职业、资质或亲历。

### 7.3 表单字段组

#### A. 基本定位

- `creator_id` / `brand_id`；
- 账号或创作者展示名；
- 创作者类型与主要内容类型；
- 专业领域和真实资质；
- 账号定位、栏目和发布目的；
- 主要平台与更新频率。

#### B. 目标读者

- 核心读者；
- 读者已有知识水平；
- 主要痛点和阅读场景；
- 希望读者看完产生的认知、情绪或行动。

#### C. 表达风格

- 正式度；
- 亲密度；
- 幽默度；
- 锐度；
- 情绪强度；
- 技术密度；
- 个人叙事密度；
- 证据密度；
- 行动建议强度；
- 第一/第二/第三人称偏好；
- emoji、感叹号、反问、破折号等使用偏好。

上述维度建议采用有限档位加文字说明，不能只留下无上下文的数字。

#### D. 结构偏好

- 常用开场方式；
- 论证骨架或叙事顺序；
- 小标题偏好；
- 案例、数据、引语和类比的使用习惯；
- 反方与边界的呈现方式；
- 结尾动作；
- 常用篇幅和移动端段落长度。

#### E. 语言资产与禁区

- 喜欢的词汇、句式和表达动作；
- 明确禁用词；
- 不希望出现的套话、标题结构和情绪；
- 不可模仿的他人标志性表达；
- 敏感议题和需要转人工确认的领域。

#### F. 真实性与身份边界

- 哪些第一人称经历有真实材料可支持；
- 哪些身份、职业、资质或使用经历不得代写或推断；
- 是否允许使用“我认为”“我们观察到”等表述；
- 亲测、访谈、业内消息和读者反馈分别需要什么证据；
- 合作、赞助、试用权益和利益关系的披露规则。

#### G. 风格样本

- 用户自有文章、链接或文件；
- 用户获授权分析的样本；
- 公开文章仅作抽象风格分析的链接；
- 每个样本喜欢什么、不喜欢什么、允许学习什么；
- 样本权利状态、日期和置信度；
- 禁止把整篇外部文章保存进 profile。

#### H. 平台差异

- 公众号、小红书、今日头条等平台的语气差异；
- 标题、段落、emoji、标签和配图偏好；
- 哪些风格特征必须跨平台保持；
- 哪些特征可以由平台适配器调整。

#### I. 确认与存储

- 表单状态：`draft / submitted / confirmed / superseded`；
- 填写人或确认人；
- 创建和更新时间；
- 对应 profile 版本；
- 同意保存的抽象风格信息范围；
- 是否允许后续用新作品提出增量更新建议。

表单不应收集完成风格建档所不必要的身份证号、住址等敏感个人信息。

### 7.4 表单处理流程

```text
选择快速/完整/增量模式
  → 生成或加载表单
  → 创作者填写并保存草稿
  → 结构校验
  → 可选：分析历史作品
  → 对比“主动填写”与“语料观察”
  → 展示一致项、冲突项和低置信项
  → 创作者逐项确认
  → 生成 profile 新版本
  → 写入版本历史和审计记录
```

优先级建议：

1. 创作者本次明确确认的选择；
2. 创作者此前明确确认的稳定偏好；
3. 多个自有样本反复出现的稳定特征；
4. 单篇样本或外部参考文章形成的暂定观察。

任何真实性、来源、版权和合规底线都不能被表单中的风格偏好覆盖。

### 7.5 强制确定性能力

后续实现必须提供以下等价能力。工具名可在开发时按仓库规范微调，但不得删减能力或降级为只靠模型记忆的对话约定：

- `mediaops.style.form.template`：按快速/完整/增量模式返回表单 schema 和初始值；
- `mediaops.style.form.save_draft`：保存未完成表单；
- `mediaops.style.form.submit`：校验并提交待确认表单；
- `mediaops.style.form.get`：读取表单及状态；
- `mediaops.profile.propose`：合并表单和语料分析，生成 profile diff；
- `mediaops.profile.confirm`：由创作者确认后生成正式新版本；
- `mediaops.profile.history` / `rollback`：查看版本和回滚。

所有写入操作都以 `brand_id` 隔离。外部样本正文不进入表单存储，只保留必要元数据和抽象特征。

### 7.6 发布阻断条件

以下任一情形存在时，阶段 3 不得验收，本次优化版本不得发布：

- 没有可调用的风格采集表单入口；
- 只能新建、不能保存和恢复未完成表单；
- 表单提交后直接覆盖正式 profile；
- 表单和历史作品分析冲突时由模型静默决定；
- 不同 `brand_id` 可以读写彼此的表单或风格版本；
- 缺少真实性边界、样本权利或创作者确认记录；
- 只有 Markdown 问卷，没有结构化 schema 和持久化状态；
- 缺少快速、完整、增量任一模式；
- 缺少确定性测试和技能级评测。

## 九、Media Gate 设计

建议定义统一停止码：

- `PROFILE_REQUIRED`
- `RESEARCH_REQUIRED`
- `REVIEW_REQUIRED`
- `UNRESOLVED_CLAIMS`
- `ORIGINALITY_REVIEW_REQUIRED`
- `LEGAL_REVIEW_REQUIRED`
- `AI_DISCLOSURE_REQUIRED`
- `PLATFORM_RULES_STALE`
- `APPROVAL_REQUIRED`
- `APPROVAL_PENDING`
- `APPROVAL_REJECTED`
- `APPROVAL_STALE`
- `PACKAGE_INPUT_MISMATCH`

核心状态迁移：

```text
intake
  → researched
  → drafted
  → reviewed
  → approved
  → packaged
```

约束：

- `drafted → reviewed`：必须有事实核查结论；
- `reviewed → approved`：必须解决或具名放行存疑项；
- `reviewed → approved`：按内容类型完成原创和法律风险路由；
- `approved → packaged`：当前内容哈希必须与审批记录一致；
- 任一已审批内容发生变化：回退 `reviewed`，审批标记为 stale；
- 平台规则过期：不得宣称“已符合最新规则”，转人工后台复查。

工作：

2. 区分硬限制和运营建议；
3. 修复中文热点聚类；
4. 增加中文热点测试集；

6. 改善微信、小红书、头条预览差异；
7. 统一所有版本号和能力描述。

验收：

- 每条硬规则可追溯到来源；
- 过期规则会要求重新核验；
- 同一中文事件的不同标题能合理聚类；
- 发布包移动后资源仍完整；
- plugin、marketplace、package、server、capabilities 版本一致。

### 阶段 5：技能评测、复核和发布

按 skill-creator 方法执行：

1. 同时运行新版与旧版基线；
2. 对事实、观点、原创、风格和门禁分别制定可验证断言；
3. 生成评测审阅页面供人工逐项复核；
4. 分析准确性、时长和 token 成本；
5. 根据反馈迭代；
6. 最后优化技能 description 的触发准确率；
7. 更新 README、marketplace 和发布说明。

建议评测规模：

- 至少 20 条触发/不触发近邻测试；
- 至少 6 个完整工作流测试；
- 所有 P0 门禁测试必须 100% 通过；
- 触发测试重点观察错误触发，而不是只看关键词命中；
- 主观写作质量必须保留人工复核，不用机械指标替代。

## 十一、不纳入本轮优化的范围

以下事项暂不进入本次实施范围：

- 微信、抖音、微博、B站等真实发布 API；
- 浏览器自动点击最终发布；
- 评论自动发送；
- 非官方平台爬虫；
- 全网查重服务；
- 对微信原创声明结果作保证；
- 用所谓 AI 检测分数作为编辑目标；
- 把媒体插件扩张成法律插件同等数量的技能集合。

上述能力如需启动，应作为独立 Gate B 项目另行审计授权、凭证、安全、平台协议和审计要求。

## 十二、最终验收标准

完成优化后应同时满足：

1. 所有发布包都能反查来源、稿件版本、profile 版本、审稿结论、审批人和审批内容哈希；
2. 未审、未批、审批后改稿均由机器硬阻断；
3. AI 披露按平台原生功能、正文提示和文件元数据分别记录；
4. 不再把固定正文句子表述成唯一法定标识形式；
5. 公众号原创观点技能与多平台总编排不存在触发冲突；
6. 插件提供快速、完整、增量三种创作者风格采集表单；
7. 表单支持草稿、提交、确认和废止状态，并能与历史作品分析做冲突比对；
8. 表单只生成待确认 profile，未经创作者确认不得覆盖正式风格；
9. 每个品牌独立保存风格表单和 profile 版本，支持 diff 和回滚；
10. 原创扫描不提供虚构原创率或平台结果保证；
11. 平台规则均有来源和核验时间；
12. 中文热点聚类通过真实中文样例测试；
13. 插件各层版本号一致；
14. 根仓库校验、媒体类型检查、单元测试和技能评测全部通过；
15. README 与代码实际能力一致，不把 Gate B 描述成已可用。

## 十三、建议执行顺序

建议将后续工作拆成独立批次：

```text
批次 A：P0 审批与事实核查门禁
  ↓
批次 B：Media Gate + content manifest + CI
  ↓
批次 C：公众号原创观点、原创复核、创作者风格表单与品牌风格模块
  ↓
批次 D：平台规则、中文聚类、发布包资源
  ↓
批次 E：技能基线对比、人工复核、版本发布
```

任何批次开始实施前都应保存基线；每个批次完成后独立运行校验和测试，不把全部改动堆到最后一次验收。

最优先执行批次 A。若先增加更多创作技能而不修复门禁，会把当前可绕过的发布链路扩散到更多入口。
