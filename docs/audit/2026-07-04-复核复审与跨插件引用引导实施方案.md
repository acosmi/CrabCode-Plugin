## 第二部分 跨插件引用引导现状审计(用户新增关切)

### 1. 需求-供给断层量化(73 插件 / 338 SKILL.md 全量扫描)

**办公文档产出**:
- 供给侧有**两套互不引用的栈**:① `crabcode-office-suite`(4 技能:crabcode-documents / crabcode-spreadsheets / crabcode-presentations / crabcode-pdf,TS 运行时);② `crabfin-cn/fin-core` 内嵌栈(`mcp__office__*` + xlsx-author/pptx-author + recalc.py)。
- 需求侧 **65 个** SKILL.md 提及文档产出(small-business 18、crabfin 16、data 5、copyright 5……),其中真正桥接到 office-suite 的**非自指引用仅 4 处**——crabcopyright-cn 3 处 + 示例插件 1 处,覆盖率约 6%。
- **crablaw-cn / cn-legal-study / crabwork-bio-research / 全部 crabwork-\* 桥接数 = 0。**
- 更深一层:法律域的关键词命中只有 2 处且偏概念性(`cn-corporate/skills/tabular-review` 的"表格化审查"是 markdown 工作底稿)。**真正的缺口不止"提了 Word 没引导",而是"交付物形态意识"缺失**——法律 memo、尽调报告、财务报表这类交付物天然应该可选 docx/xlsx 落盘,现状却全部止步于 markdown 文本。关键词统计是缺口的下界。

**深度调研**:
- **31 个** SKILL.md 提及联网调研/检索(crablaw 11、crabfin 6、sales 4、marketing 2、cn-legal-study 2……),引用统一调研能力的 = **0**。
- 无插件级 deep-research 供给方;各域自建且互不复用:crablaw `matter-deep-analysis`(文档尽调,非联网)、media-ops `trend-researcher`(热点取材)、crabwork 各域研究工作流。crablaw `cn-legal-aid/skills/research-start` 是手工检索规划模板,明确境内合规渠道约束。

**机制层**:
- 73 份 plugin.json **零依赖声明**;SKILL.md frontmatter 无依赖字段(最接近的 `compatibility` 自由文本仅 2 处);跨技能引用唯一形态是"散文按名提及"。
- 全库唯一成熟范式:`crabcopyright-cn/apply-core/skills/apply-manager/SKILL.md:29-34`——**开场依赖自检 + 缺失时引导 `/plugin` 安装**(07-01 P1-5 落地产物,已验证)。

### 2. 新发现缺陷:crabfin-cn 办公栈三重断链(本次引用引导审计的直接战果)

crabfin-cn **没有 `.mcp.json`**,但其技能引用了三类不存在的设施:
1. `recalc.py` **全仓不存在**(Glob 零命中),而 dcf-model/lbo-model/three-statement-model 共 14+ 处指挥模型"交付前必须跑 `python recalc.py`"(dcf-model/SKILL.md:805-846、lbo-model/SKILL.md:36);
2. `lbo-model/SKILL.md:167` 写死上游容器挂载路径 `/mnt/skills/public/xlsx/recalc.py`,CrabCode 环境无此挂载;
3. xlsx-author/pptx-author(SKILL.md:9,43-44)以 `mcp__office__excel_*` / `mcp__office__powerpoint_*` 为首选"live"路线,该 MCP server 从未声明,工具不存在。

即:财务建模技能的 headless 路线(recalc.py)和 live 路线(mcp__office__)**双双断链**,是上游 Anthropic 容器环境的遗产未清洗。这印证了用户直觉:引用引导不只是"锦上添花的路由",需求侧现状已经在引用不存在的能力。

### 3. 运行时机制证据(D:\crabcode 源码,决定引导文案写在哪才有效)

| 机制 | 证据 | 对方案的约束 |
