## 1. 背景

CrabCode 1.0.16 为「插件源隔离」与「64 字符 API 名限制」,把插件 server namespace 与带点工具名哈希为 wire name。这引入两个纯宿主侧缺口,插件侧无法自愈:

- **P0-C 发现缺口**:`mediaops.originality.scan` 在模型工具面不再保持原始名,ToolSearch 未把权威 `mcpInfo.serverName`/`toolName` 作为逻辑别名索引,Skill 按文档名搜索/选择时即使服务已连接也可能找不到哈希后的实际工具(§4.2)。
- **P0-D 激活缺口**:冷 worker 首轮故意使用空 MCP 集以避免远程 OAuth 拉长首 token 延迟;该策略对 required-local sidecar 不合适,导致首轮拿不到本应有界即可用的核心工具(§7.4)。

实施方案 §4.2 已定性:「这属于宿主侧的系统性兼容缺口,不应由每个插件把哈希 wire name 写入 Skill。」——即两项都必须在宿主修,插件侧不得以写死 wire name 等手段绕过。

## 2. P0-C:ToolSearch 逻辑工具名别名

### 2.1 现状索引与缺口

宿主 ToolSearch 当前主要索引:wire tool name、tool description、可选 `searchHint`;但**未**把权威 `tool.mcpInfo.serverName` 与 `tool.mcpInfo.toolName` 作为逻辑别名。

涉及宿主文件(实施方案 §4.2):

- `src/services/mcp/mcpStringUtils.ts`
- `src/services/mcp/connectionAndFetch.ts`
- `src/tools/ToolSearchTool/ToolSearchTool.ts`

### 2.2 两层名称模型(实施方案 §7.3)

```text
wire name: mcp__p_<hash>__h_<hash>
logical aliases:
  plugin:crabcode-media-ops:mediaops     # 完整限定:plugin:<pluginName>:<serverName>
  mediaops                               # 裸 serverName
  mediaops.originality.scan              # serverName.toolName(带点原始名)
  originality.scan                       # 裸 toolName
```

四种形态都应能双向解析到唯一 wire tool;权限与实际调用仍使用权威 wire identity,别名**只服务发现**。

### 2.3 实现要求(§7.3 五条 → 可实现规格)

1. **索引扩展**:关键词索引在现有字段之外新增 `tool.mcpInfo.serverName` 与 `tool.mcpInfo.toolName`;建议在 `connectionAndFetch.ts` 工具入库处、`ToolSearchTool.ts` 查询匹配处同时接入,避免只改一端造成入库/查询不对称。
2. **精确选择**:`select:mediaops.originality.scan` 必须解析到唯一 wire tool(对应 §8.1「logical select」用例)。四种别名形态(§2.2)均需命中同一 wire tool。
3. **歧义不静默**:多插件声明同一 logical name 时,返回**歧义候选列表**(建议附各自 `plugin:<pluginName>:<serverName>` 完整限定名以便消歧),**不得静默任选**(对应 §8.1「ambiguous alias」用例)。完整限定形态 `plugin:<pluginName>:<serverName>[.<toolName>]` 应始终唯一、可绕过歧义。
4. **权限不走别名**:别名仅用于发现/检索排序;授权判定与工具调用继续使用权威 wire identity,不得因别名解析放宽任何 principal/role 检查。
5. **别名规范化**:在 `mcpStringUtils.ts` 增加 serverName/toolName → 逻辑别名的规范化函数(大小写、分隔符、`plugin:` 前缀的统一),入库与查询共用同一函数,保证对称。

### 2.4 回归测试清单(交由宿主库落地)

- 插件哈希 server:声明含点工具名的插件 server,验证四种别名形态各自解析到同一 wire tool;
- 带点 tool:`serverName.toolName` 与裸 `toolName` 两路命中一致;
- 冲突别名:两插件同 logical name → 返回歧义候选、不静默选择;完整限定名唯一解析;
- 权限不放宽:别名命中后,匿名/缺角色调用仍分别 fail 到 `AUTHENTICATION_REQUIRED`/`AUTHORIZATION_DENIED`(与插件侧身份矩阵 §8.3 对齐);
- 索引对称:入库端与查询端使用同一规范化函数的属性测试。

## 3. P0-D:required-local 首轮有界初始化

### 3.1 现状与缺口

CrabCode 冷 worker 首轮故意使用空 MCP 集,以避免远程 OAuth 把首 token 延迟拉长(§7.4)。该策略对 required-local sidecar 不合适:此类 sidecar 无远程认证、可离线秒级启动,首轮就应在有界时间内可用。

### 3.2 分流策略(实施方案 §7.4)

| 服务类别 | 首轮策略 |
| --- | --- |
| `required + local + config-complete` | **首轮有界等待**,核心工具在预算内出现 |
| optional local | 后台初始化或用户显式启用 |
| remote / OAuth | 继续后台初始化(维持现状,避免拉长首 token) |
| 首轮超时 | 暴露明确 lifecycle 状态,**不让 Skill 猜测** |

### 3.3 前提条件(插件侧已满足)

§7.4 要求 required-local 满足「无启动时安装、离线可启动、时间有界」方可纳入首轮有界等待。media-ops 0.4.1 已全部满足(§4 详列):`.mcp.json` 直接执行自包含 `dist/server.js`,`start` 脚本不含任何 `install`,禁网/最小 PATH/无 `node_modules` 环境冷启动 smoke 通过。

### 3.4 超时预算建议:10s

以插件侧实测冷启动为依据(禁网、仅复制发行物、最小 PATH `initialize → tools/list → capabilities` 全链路):

| 指标 | 实测值 |
| --- | --- |
| 冷启动 smoke 全链路耗时 | **754 – 2215 ms**(首次 2215ms 含 bun 首解析,复次 754–803ms) |
| `tools/list` 返回工具数 | **38** |

建议宿主首轮有界等待预算取 **10s**:约为最坏实测(2.2s)的 4.5×,覆盖慢盘/首解析/杀软扫描等冷机放大,同时远小于会被用户感知为「卡住」的量级。该值应可由宿主配置覆盖。

### 3.5 超时降级契约

首轮在预算内未出现核心工具时,宿主**不得**静默留空 MCP 集让 Skill 误判,而应暴露明确 lifecycle 状态。该状态需能被插件侧预检契约映射为停止码(见 §4.3):

- 服务进程未起 / 首轮未激活 → `MCP_INACTIVE`;
- 进程起了但握手/首轮预算内未就绪 → `MCP_START_FAILED`;
- 工具集非空但目标逻辑名解析不到 → `MCP_TOOL_UNDISCOVERABLE`(与 P0-C 别名解析结果联动)。

### 3.6 回归测试(§8.1「first-turn required local」)

- 冷 worker + 离线发行包:首轮在有界时间内出现核心工具(38 个,或至少 `mediaops.capabilities`/`mediaops.doctor` 等预检工具);
- 预算内未就绪:返回明确 lifecycle 状态,可映射到上述停止码,Skill 侧走「未执行门禁」路径而非伪装完成;
- `required + local` 与 `optional local` 分流互不影响:optional-local 不被强拉进首轮阻塞。

## 4. 插件侧已就绪的下游契约(宿主补丁可依赖的既成事实)

media-ops 0.4.1(本仓已合并)为本规格提供稳定接口,宿主补丁可直接依赖:

### 4.1 required 声明

`plugins/crabcode-media-ops/.crabcode-plugin/plugin.json` 已声明 `"requiredMcpServers": ["mediaops"]`;`.mcp.json` 的 `mcpServers.mediaops` 真实存在且经本仓 `validate-mcp-contract` 校验(名称存在性、无安装器、产物路径 `${CRABCODE_PLUGIN_ROOT}/dist/server.js` 存在、三处版本一致)。宿主 P0-D 分流可信任该声明。

### 4.2 自包含发行物与冷启动

- `.mcp.json` 直接执行 `bun --no-env-file ... ${CRABCODE_PLUGIN_ROOT}/dist/server.js`,**无启动时安装**;
- 重型 delivery QA 依赖(Playwright/axe/Chromium)惰性探测,缺失时返回 `DEPENDENCY_NOT_READY`,**不影响** MCP `initialize`/`tools/list`/基础工具;
- 冷启动 smoke(`check:distribution`)在 CI(Linux 权威)与本地均通过,数据见 §3.4。

### 4.3 停止码契约(供 P0-D 超时降级复用)

插件预检契约(`media-core/PRACTICE.md`《运行前预检与停止码》)已定义 7 个停止码,宿主 lifecycle 状态应可映射到其中:`MCP_INACTIVE`、`MCP_START_FAILED`、`MCP_TOOL_UNDISCOVERABLE`、`AUTHENTICATION_REQUIRED`、`ROLE_REQUIRED`、`DEPENDENCY_NOT_READY`、`GATE_NOT_EXECUTED`。宿主只需暴露可判别的 lifecycle 原因,Skill 侧即可产出正确停止码而非伪装完成。

## 7. 回滚(实施方案 §9.3,宿主部分)

- 宿主 ToolSearch 别名改动**不写业务数据**,可独立回滚,不影响插件或数据;

- 回滚后功能可停止,但**不得把未完成门禁显示为完成**。

- 本规格**不改动**任何宿主代码;`D:\CrabClawApp` 陈旧检出**不是**落地对象(§0.1);
