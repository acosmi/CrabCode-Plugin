## 0. 一页速览(TL;DR)

- **问题**:插件的 CRM 连接器仍默认绑 **HubSpot**(海外 SaaS,数据存境外)。两重不合国内小微:①数据出境是 **PIPL** 负担;②HubSpot 的"销售管道(deal/pipeline)"范式不合中国小微——小微的客户关系**住在微信里**,主流范式是 **SCRM(社会化 CRM)**。
- **决策(已与用户确认)**:选定四家国内 CRM——**企业微信 / 钉钉 / 飞书 / 有赞**;**HubSpot 不拆除,降级**为"外贸/跨境场景可选"。
- **打法**:先做**话术层去默认化**(零连接器代码,当下即让技能不再默认推境外 CRM),再分平台落**连接器 + CRM 能力抽象层**。
- **不变量**:插件"工具无关 / 说能力不推销"哲学保持;PIPL 与审批闸门护栏平台中性化后**保留**;易变的 API 端点/scope/字段/套餐门槛**不硬编码**,以各开放平台当年官方文档为准;子代理只写码、git 主控独占、独立分支、不碰 main、validate 实跑绿;版本停 0.2.0,随家族协调升 minor(0.3.0)。

## 1. 背景与触发

crawork 系列由上游 "Claude for Work" 套件整体移植改名而来,移植时 **MCP 端点原样照搬**——上游面向美国中小企业,CRM 默认绑 HubSpot 官方 MCP(`https://mcp.hubspot.com/anthropic`,见 `.mcp.json`)。

6-23 总审计已把连接器替换列为工程主线,crawork 是阶段 1 试点。此前完成的工作:
- **收款/IM/文档层**已国内化:支付宝、钉钉、飞书、腾讯文档(官方 MCP 直连)。
- **中国法化**(姊妹档)已把 7 个法律域的技能正文改到中国口径。
- CRM 一直挂着"选型待定,暂保留 HubSpot"的过渡口径(见 `CONNECTORS.md` 第 19 行)。

本次即收掉这条尾巴:**把 CRM 从"默认 HubSpot"迁到国内四家,HubSpot 降级保留。**

**为什么现在值得做(第一性原理,非跟风):**
1. **合规实质收益**:CRM 存的是姓名/电话/邮箱/消费记录等个人信息。境内 CRM = 数据本地化,直接缓解 PIPL 数据出境合规面;境外 CRM 对小微是持续的合规不确定性。
2. **范式错配**:HubSpot 内核是 B2B 销售漏斗;中国小微的客户关系发生在**微信私域**(加微信、朋友圈、客户群、复购)。硬套销售管道,老板用不起来。正确范式是 **SCRM**。
3. **贴合既有哲学**:插件本就声明**工具无关**(`CONNECTORS.md` 以类别而非产品描述工作流),onboarding 也是"说能力、问你在用啥、不推销"。多 CRM 路由天然契合这套框架,不是外挂。

## 2. 与前序审计的对齐与**有意分叉**(架构漂移审查)

6-23 审计给 CRM 开的候选是 **销售易 / 纷享销客**(见该档 CRM 行与遗留问题第 4 条:"销售易 vs 纷享销客——还是先不做 CRM、HubSpot 直接移除?")。

本方案**有意分叉**,理由须明确记录(避免"悄悄改方向"):

| 维度 | 6-23 审计候选(销售易/纷享销客) | 本方案候选(企微/钉钉/飞书/有赞) |
|---|---|---|
| 选型逻辑 | **同形替换**——找一个"像 HubSpot 的" B2B 管道 CRM | **贴合小微实际**——客户在微信/私域/零售,按渠道路由 |
| 目标客群 | 中大型、有专职销售团队 | **小微/个体**(本插件的真实用户) |
| 形态 | 销售管道 CRM | SCRM(企微)+ 套件内 CRM(钉钉/飞书)+ 零售会员(有赞) |
| 接入 | 销售易 MCP 内化不可外连;纷享销客平台绑定 | 见 §3 各家 |

**结论**:销售易/纷享销客对"有销售团队的 B2B"仍成立,**留作未来 B2B 适配器**(§5 抽象层可容纳),但**不作小微默认**。本次四家更贴合插件的真实客群。这是对前序审计"待定选型"问号的**收敛**,不是推翻其工程主线。

## 3. 选型决策(已确认)

### 3.1 选定四家

| CRM | 定位 / 适合谁 | 形态 & 关键 API | 成本 | 接入性质(以当年开放平台文档为准) |
|---|---|---|---|---|
| **企业微信**(默认推荐) | 客户在微信里的绝大多数小微——私域、复购、加微信 | SCRM:外部联系人 / 客户群 / 客户朋友圈,**客户联系 API** | 免费 | 官方无完整 MCP(社区仅 webhook)→ **需自建 wrapper**(阶段 3 类) |
| **钉钉** | 已用钉钉办公的 | 套件内**客户管理 / 智能 CRM** 模块 | 随钉钉 | **已连**(IM/日程 MCP)→ 需核实/扩展 CRM 对象 scope |
| **飞书** | 已用飞书办公的 | 套件内 **飞书 CRM / 销售线索** | 随飞书 | **已连**(IM/日历/文档 MCP)→ 需核实/扩展 CRM scope |
| **有赞** | 零售门店 / 电商 / 会员制(如五金店王芳) | 零售:**会员 / 订单 / 客户 / 导购**(deal≈订单) | 有免费/低价档 | 有开放平台 API → **需自建 wrapper** |

### 3.2 HubSpot 降级(用户确认)

- **不拆除,降级**:从"默认 CRM"降为 **"外贸/跨境 B2B 场景可选"**。
- 理由:部分做外贸的小微在合法使用它(海外买家、英文管道);插件本是平台无关框架,保留合法路径比删掉更对。
- 落地:`.mcp.json` 保留 hubspot 条目;技能话术中 HubSpot 从"默认数据源"改为"若你在用 HubSpot(常见于外贸)……"的可选分支;onboarding 不再首推。

### 3.3 落选说明(记录以免反复)

- **微盟**:与有赞在"智慧零售 SCRM"高度重叠,先落一家(有赞),避免同形冗余;需要时再作为有赞的同类适配器补入。
- **销售易 / 纷享销客**:企业级偏重,见 §2;留作未来 B2B 适配器,非小微默认。
- **Zoho 等境外系**:与 HubSpot 同属境外,不额外引入。

## 4. 分渠道路由逻辑(smb-onboard / smb-router 将据此实现)

onboarding 的推荐不是"推某一家",而是**问清老板画像 → 路由到对的那家**:

| 老板画像 | 推荐 CRM | 为什么 |
|---|---|---|
| 客户主要在微信里(私域、复购、加微信) | **企业微信** | SCRM,客户联系 API,免费,覆盖面最广 |
| 已用钉钉办公 | **钉钉客户管理** | 零新连接器(套件已连),套件内闭环 |
| 已用飞书办公 | **飞书 CRM / 销售线索** | 零新连接器(套件已连),套件内闭环 |
| 零售门店 / 电商 / 会员制 | **有赞** | 会员+订单+导购,零售形态原生 |
| 外贸 / 跨境 B2B | HubSpot(可选保留) | 海外买家、英文管道 |

保持既有话术纪律:**说 CrabCode 连上后能做什么,而非介绍该平台是什么**;老板决定,不推销。

## 5. 架构:从"HubSpot 硬编码"到"CRM 能力抽象层 + 适配器"

### 5.1 现状 gap

`CONNECTORS.md` 声明插件**工具无关**(用 `~~类别` 占位),但**技能正文全程硬编码 "HubSpot"**、并直接引用 `hubspot-scoring.md` / `hubspot-fields.md` / `hubspot-staging.md`。声明与实现脱节。

### 5.2 目标:中性能力概念 + 每平台适配器

抽象出平台中性的 CRM 能力词汇——**客户 / 线索 / 商机 / 客服工单 / 营销活动**——技能只谈这些概念;每个平台配一份 **adapter 引用**,描述"中性概念 ↔ 该平台对象/字段/API"的映射与该平台的能力边界。

**为什么不能简单"HubSpot 改名企微"**:四家数据模型根本不同——
- HubSpot:deal / pipeline / lifecyclestage(管道)
- 企业微信:外部联系人 / 客户群 / SCRM 标签(**无"商机管道"**;"打分"要落到互动/标签而非 deal 金额)
- 有赞:会员 / 订单 / 客户(**deal ≈ 订单**,营销≈会员营销/优惠券)
- 钉钉/飞书:套件内 CRM 对象(介于两者之间)

直接字符串替换会产出跑不通的技能。必须走"中性概念 + adapter 映射"。

### 5.3 YAGNI 边界(不过度工程)

小微通常**只连一家** CRM,不会四家同开。所以插件**不需要同时跑四家的运行时**,只需要:
1. onboarding 把老板**路由到对的那一家**;
2. 每个技能按**当前已连接的那家** adapter 取数、并在缺失时优雅降级(既有降级模式已在,复用)。

抽象层是"一份中性技能正文 + N 份 adapter 引用",**不是**一个运行时多路复用引擎。够用即止。

## 7. 实施分期(批次计划,逐段待放行)

**Phase 0 —— 落档(本档)** ✅ 本次完成。

**Phase 1 —— 话术层去默认化(零连接器代码,当下即生效)**
目标:让技能**立刻停止默认推境外 CRM**,即便连接器还没建。
- CRM 提法中性化:"你的客户系统 / CRM",四家作为菜单,**企业微信为默认推荐**,HubSpot 降为"外贸可选"。
- 改:`CONNECTORS.md`(CRM 行)、`README.md`(相关列)、`smb-onboard`(happy-path 里阿伟的 Pipedrive 例子改写为国内情景)、`onboard-checklist.md`、`smb-router`(路由/数据源表)。
- 保留:所有审批闸门与 PIPL 护栏(仅把"HubSpot"换成中性表述)。
- 交付:一次批次提交;`bun run validate` + `bun test ./tests/` 实跑绿。

**Phase 2 —— CRM 能力抽象层 + 适配器(需真连接器)**
- **2a 抽象层骨架 + 最省力适配器验证**:先建中性能力引用,拿**钉钉或飞书 CRM**(复用现有连接器,最省力)跑通一条链路,验证抽象是否成立——**先用最便宜的 adapter 证明抽象,再上贵的**。
- **2b 企业微信 SCRM 自建 wrapper**:覆盖面最广、价值最高,但需自建(客户联系 API)。含 `.mcp.json` 新条目 + 企微 adapter 引用 + 技能接线。
- **2c 有赞零售适配器**:会员/订单/客户,自建 wrapper。
- 每个适配器:MCP/接入配置 + adapter 字段映射引用 + 技能接线;**API 端点/scope/字段/套餐门槛一律以各开放平台当年官方文档为准,不硬编码**。
- 顺序可调:若市场优先级要求先上企微,可把 2b 提前(代价是先啃自建);默认建议 2a→2b→2c 以先低成本验证抽象。

**Phase 3 —— HubSpot 降级收尾 + 全量校验**
- HubSpot 作为"外贸可选"适配器归位;确保所有技能按"已连接的那家"取数并优雅降级。
- 全量 `validate-all`(7 linter)+ `bun test`;毁灭性交付复核;主控显式 merge。

## 8. 边界与硬约束(实施时必须守)

1. **平台无关框架 & 说能力不推销**:既有哲学,保持;新增四家不破坏它。
2. **PIPL 护栏保留(平台中性化)**:`crm-cleanup` 已有的"合法取得/授权/最小必要/合并前逐条确认/跨境与批量导出移交 `crablaw-cn:data-activity-triage`"必须**原样保留**,只把平台名中性化。SCRM(企微)涉客户朋友圈/客户群,个人信息面更广,护栏只增不减。
3. **审批闸门保留**:绝不代发消息/代改商机阶段/代删记录/写入前不 announce——这些闸门平台中性,逐平台重表述后保留。
4. **不硬编码易变项**:各平台 API、字段名、scope、套餐门槛(如原 HubSpot Marketing Hub Professional 之类)随平台政策变动,一律指向"当年官方开放平台文档核实",不写死。
5. **交付纪律**:子代理只写码;**git 由主控独占**;独立分支、**绝不碰 main**;只 stage 具体路径;`build/validate/test` 实跑绿再提交;毁灭性复核过一遍(边界/错误路径/契约/平台差异/安全)。
6. **红线一致性**:本次是连接器/话术层改动,不触碰法化任务的税务/法律/人力红线;两档互不回改。
7. **版本**:crawork 停 **0.2.0**(现与家族对齐);本次改动量按语义化版本属 minor,建议**随家族下次协调发布统一升 0.3.0**,不单独抢跑,以免 marketplace 版本错位。
