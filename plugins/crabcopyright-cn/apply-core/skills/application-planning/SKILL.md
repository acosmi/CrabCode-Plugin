# 软著申请板块规划（第 0 步）

在做任何材料之前,先回答一个结构性问题:**这套代码应该登记成几个软著、怎么拆。**
拆得好,一份代码换多个真实证书,边界还和产品对得上;拆得差,要么浪费多证书价值,
要么各块凑不满、代码重叠被查重驳回。**先读**
`${CRABCODE_PLUGIN_ROOT}/apply-core/GUIDE.md`(尤其 §3 源码 60 页、§4 说明书功能对应红线)。

## 第一性原理：为什么拆、拆的边界在哪

一个软著的原子单位 = 一个由(**全称 + 版本号**)标识的「软件」,证书保护的是这个被命名
软件的表达。据此推导:

**拆多个的收益**
1. 每个证书独立保护一个**可命名、可交付、可单独售卖/授权**的单元,边界与真实产品对齐。
2. 软著数量是很多资质与激励的计数项——高新技术企业认定、科技型中小企业、政府补贴、
   招投标加分、融资尽调。**多个真实证书 > 一个大证书。**
3. 版本迭代、授权、转让都能按单元独立进行。

**拆的成本与红线**
1. 每个申请要**独立的 60 页源码 + 60 页说明书**,且**代码必须与该软件功能真实对应**。
   拆出来的每块得有足够、**不与其它申请重叠**的实质代码,否则跨申请代码雷同 = 注水/查重驳回。
2. 公共库、design-system、公用 SDK、脚手架不宜单独成软著,也**不能在多个申请里重复充数**;
   应归到最相关的那一个软件,或只在一个申请里体现。
3. 拆太细 → 每块凑不满 ≥3000 行有效代码、说明书功能单薄;拆太粗 → 浪费多证书价值、
   名称与交付物对不上。

**边界判据(沿真实接缝切)**:可独立命名、可独立部署/交付、功能内聚、有自己的用户界面
或对外接口、有足够不重叠的实质代码。天然接缝通常是:monorepo 的 `apps/*`、`services/*`、
独立部署单元、前端多端(管理后台 / 商户端 / 用户端 / 小程序)、后端服务、独立 SDK。

## 步骤

### 1. 探测仓库结构
用 Glob/Grep/Read 识别候选拆分单元:
- workspace 配置:`pnpm-workspace.yaml`、`package.json` 的 `workspaces`、`turbo.json`、`nx.json`、`lerna.json`。
- 目录布局:`apps/*`、`packages/*`、`services/*`、`frontend`/`backend`、`server`/`web`/`admin`/`miniprogram`。
- 多语言子项目:`go.mod`、`Cargo.toml`(workspace)、`pom.xml`、`pyproject.toml`、独立 `Dockerfile`/部署清单。
- 前端框架与端:React/Vue/Next/Nuxt/Angular/Taro/uni-app 等,判断有几个"端"。

### 2. 估算每块的有效代码量
统计各候选单元的源码文件数/行数(排除 `node_modules`/`dist`/`build`/生成物/锁文件)。
**≥3000 行有效代码**是能舒适撑起 60 页的经验线;不足的单元考虑合并到相邻单元。

### 3. 结合目标数量给出拆分
读取用户想要的申请数量 N(缺则用 AskUserQuestion 问,并说明"能拆几个"的上限受有效代码量约束)。
给出推荐拆分——**可能等于 N,也可能建议调整 N** 并讲清理由(如"第 3 块只有 800 行,
建议并入第 1 块,否则说明书会单薄")。

### 4. 定位每个申请的截图页面与开发环境地址
说明书要"图文并茂展示主要功能"、审查员据此判断功能真实且与代码对应。所以每个申请要挑
**覆盖其核心功能路径**、且能在开发环境实际打开的页面:
- **取路由表**:React Router 的 `<Route path>`、Vue Router 的 `routes`、Next.js 的
  `app/`或`pages/`目录结构、Nuxt/Angular 路由。
- **取 dev 端口**:`vite.config.*` 的 `server.port`、`package.json` 的 `dev` 脚本、
  `next.config.*`、`.env` 的 `PORT`、`vue.config.*`。
- **拼访问地址**:`http://localhost:<port>/<route>`。
- **选页面**:登录页 + 主界面/仪表盘 + 每个核心功能模块的代表页(增删改查主页),
  覆盖说明书要讲的功能点。

### 5. 初始化各申请的 manifest

拆分与命名经用户确认后,为**每个申请**创建 `outputs/<申请名>/manifest.json`
(结构见 `${CRABCODE_PLUGIN_ROOT}/apply-core/MANIFEST.md`),写入 `application_name`、
`software`(全称/简称/版本号)、`source.dirs`(该申请归属的源码目录)、
`manual.screenshot_plan`(上面的截图取证清单),`steps.application-planning` 置 `done`。
后续工序一律从 manifest 取参数,不靠口头交接。

## 成功标准

- [ ] 识别出仓库的真实拆分接缝,而非机械按目录数拆
- [ ] 每个建议申请都有足够(≥3000 行经验线)且不重叠的源码,并说明公共代码归属
- [ ] 每个申请有截图取证清单:页面 + 路由 + 开发环境地址 + 对应功能点
- [ ] 若目标数量与代码量矛盾,明确给出调整建议与理由
- [ ] 用户确认后,每个申请的 `outputs/<申请名>/manifest.json` 已初始化

## 检查点

规划表输出后暂停,与用户确认拆分方案和每个软件的命名,再回软著申请管家按申请逐个推进。
