# 软著材料清单核查

按用户的**申请人类型**与**开发方式**,输出这一具体情形所需的完整材料清单,
并逐项与用户确认状态。目的是在动手做鉴别材料前,先把"要交什么、缺什么、
特殊情形要补什么"一次说清,避免做到一半才发现缺许可证明或合同。

**先读** `${CRABCODE_PLUGIN_ROOT}/apply-core/GUIDE.md` §1、§2——身份证明与特殊情形补充
材料以它为准。若担心平台要求有更新,用 WebSearch/WebFetch 核对官网所需文件页作为基准。

## 步骤

1. 确认 `$APPLICANT_TYPE`(个人/企业/事业单位)与开发方式(独立/合作/委托/下达任务/职务/继受取得/在原有软件上二次开发)。缺则用 AskUserQuestion 问清——**开发方式直接决定要不要补合同或许可证明**。
2. 组装清单:基础三大件(申请表、源程序、文档)＋ 对应身份证明 ＋ 该开发方式的补充证明(见 GUIDE.md §2.2 表)。
3. 逐项与用户核对状态。

## 输出格式

```
## 软著申请材料清单 · ${SOFTWARE_NAME} ${VERSION} · ${APPLICANT_TYPE}

### 基础材料（必交）
- [状态] 软件著作权登记申请表
- [状态] 源程序鉴别材料（前30+后30页）
- [状态] 文档鉴别材料（说明书/手册,择一）
- [状态] 身份证明（个人:身份证正反面 / 企业:营业执照副本 / ...）

### 本情形补充材料（据开发方式）
- [状态] 合作开发合同 / 委托开发合同 / 原著作权人许可证明 / 权利转让合同 ...

### 说明
- 提示费用(2017-04-01 起官方免费)、审限(受理起60日内)、2026 新版申请表要点
```

状态用:✅ 已有 / ❌ 缺失需补 / ⚠️ 待确认。

## 成功标准

- [ ] 所有必交项状态为 ✅,或用户已知晓缺失需补
- [ ] 依开发方式列出了正确的补充材料(如二次开发列出"原著作权人许可证明")
- [ ] 身份证明与 `$APPLICANT_TYPE` 匹配
- [ ] 存在 ❌/⚠️ 项时,清单末尾明确列出待办

## 检查点

清单输出后暂停,等用户确认已有/缺失,再回到软著申请管家推进下一步。
