# 0.4 升级与运行迁移

本说明面向从 0.3.x 或早期 0.4 候选版升级的安装。0.4 改变了持久化、身份、研究输入和交付验收契约；旧调用方若继续提交原字段，应当收到明确错误，而不是被兼容层静默放行。

## 1. 先备份数据目录

确认实际 `MEDIAOPS_DATA_DIR`（CrabCode 安装通常指向 `${CRABCODE_PLUGIN_DATA}`），停止仍在写入旧数据的 media-ops 进程，并备份整个目录。不要只备份 `*.jsonl`，profile 版本、冻结交付物和素材也属于审计链的一部分。

## 2. JSONL 到 SQLite

新工作流记录写入数据目录下的 `mediaops.sqlite`，使用 SQLite WAL 事务、`FULL` 同步、集合哈希链、实体 CAS 和短期租约。旧 `<collection>.jsonl` 与 `<collection>.head.json` 不应手工拼接、截断或删除。

首次读取或写入某个旧集合时，插件会：

1. 逐行解析旧 JSONL；
2. 校验每条记录哈希、前序哈希、尾部 head 与记录数；
3. 在单个事务中导入 SQLite；
4. 写入该集合的迁移完成标记，避免重复导入。

任何坏行、断链、尾部截断或 head 不一致都会返回 `STORAGE_CORRUPT`。此时保留原文件和备份，先调查损坏原因；不要通过删除最后一行或伪造 head 来让迁移“通过”。

没有历史哈希的旧 JSONL 也默认拒绝。只有完成离线备份、人工确认来源并接受“迁移前历史无法认证”后，才可为一次迁移临时设置：

```text
MEDIAOPS_ALLOW_UNVERIFIED_LEGACY_IMPORT=I_ACCEPT_UNVERIFIED_HISTORY
```

导入完成后立即取消该变量。插件会从导入字节重建新哈希链，但新链只能证明导入后的连续性，不能追溯证明旧记录从未被改写；正式审计必须保留这个信任降级事实。

profile 的权威记录也迁入 SQLite，并与 `profile.version.confirmed` 审计事件同事务提交。`profiles/<brandId>/versions/` 的不可变 JSON 和 `current.json` 现在是提交后原子物化的派生副本；物化失败会返回持久化告警，读取时从 SQLite 修复，不能把派生文件当成高于数据库的事实源。

## 3. 配置可信 principal

0.4 不再把工具参数里的 `savedBy`、`completedBy`、`reviewedBy`、`approvedBy` 等姓名当成身份。所有变更工具都要求以下两种可信来源之一：

- MCP OAuth：`authInfo.extra.sub`（或 `subject`）、issuer，以及 scopes/roles；
- 明确信任的宿主 principal：

  ```text
  MEDIAOPS_IDENTITY_MODE=host-principal
  MEDIAOPS_TRUSTED_PRINCIPAL_ID=<稳定主体 ID>
  MEDIAOPS_TRUSTED_PRINCIPAL_ISSUER=<宿主发行者 ID>
  MEDIAOPS_TRUSTED_PRINCIPAL_ROLES=<逗号分隔角色>
  ```

宿主 principal 是部署者的信任声明，不是插件登录系统。一个进程中的同一 `issuer:principalId` 不能同时满足请求者/批准者或作者/独立核查者的跨人隔离；正式多人流程需要宿主为不同用户注入不同 subject。

按实际任务授予最小角色：

| 工作 | 角色 |
| --- | --- |
| 保存内容、登记参考、采集页面 | `author`、`reference_curator`、`researcher` |
| 完成研究、原创扫描/人工复核、编辑复核 | `fact_checker`、`originality_scanner`、`originality_reviewer`、`editorial_reviewer` |
| 生成交付、独立视觉复核 | `renderer`、`delivery_reviewer` |
| 风格表单/导入与最终 profile 确认 | `profile_editor`、`profile_approver` |
| 请求审批、批准、打包 | `approval_requester`、`approver`、`publisher` |

可用 `mediaops:<role>` 形式表达作用域角色；`mediaops:*`/`*` 虽被服务端识别，但不应用于正式多人职责分离。

迁移时先调用 `mediaops.capabilities` 检查 `authenticatedActorIdentity`、`actorIdentityAssurance` 和 `actorRoles`。未配置身份时，变更工具返回 `AUTHENTICATION_REQUIRED` 是预期的安全行为。

## 4. 更新研究来源输入

删除旧请求中的 `sourceTier`、`isPrimary` 和 `originPublisher`。它们不再是可接受的来源输入。

每个 `sources[]` 项改为提交：

- `sourceKey`、服务端生成的 `captureId`、`title`、`publisher`；
- 可选 `creator`、`publishedAt`、`rightsOrTerms`；
- 必填 `assessment.publisherType`；
- 必填 `assessment.sourceFunction`；
- 必填 `assessment.originRelationship`；
- 能在 capture 快照中原样定位的 `assessment.basisExcerpt`；
- 至少 20 字符的 `assessment.classificationRationale`。

服务端验证评估字段之间是否相容，再推导 `sourceTier`/`isPrimary`。独立来源组按组织域、责任发布者、页面身份、精确快照和近似内容聚类；调用方不能指定或拆分 `independenceGroup`。

原始记录、第一方声明和官方解释还必须通过发布者主机身份验证。内置机构域仅覆盖受限的政府/法院/学术规则；其他官方主机由部署者配置逗号分隔的 `MEDIAOPS_TRUSTED_SOURCE_HOSTS`。精确主机只匹配自身；`*.agency.example.cn` 只匹配子域、不匹配裸域，过宽或非法项会使研究保持 `action_required`。输出中的 `publisherIdentityRule` 保存实际命中规则；配置型信任还保存规范化规则集哈希 `trustedHostConfigurationHash`，便于日后复验配置漂移。该配置是部署者信任声明，不替代内容证据。

完成研究后，调用公开只读 `mediaops.research.get(researchId)` 取得含服务端 `sourceId`、来源、evidenceLinks 和哈希的完整研究包。旧调用方不得继续依赖内部函数或日志拼装交接数据。

## 5. 补齐正文 statement coverage

`mediaops.editorial.review` 会把当前标题、摘要、正文、引文元数据、图片 alt/图注及披露中的全部可见句子提取为 `statements`，并要求 `statementCoverage` 逐项分类。不要在客户端复制内部哈希算法。推荐迁移后的调用流程是：

1. 对当前 revision 提交完整 claims、法律/披露信息，以及空或尚不完整的 `statementCoverage`；
2. 从 `action_required` data 读取服务端生成的 `statementLedgerHash` 和 `statements[]`（含 `statementId`、位置、正文与信号）；
3. 人工逐项分类并补齐 coverage；
4. 对同一 revision 重试；成功响应保留最终 `statementLedgerHash` 供后续交接核对。

每项覆盖必须选择一种语义：

- 使用工具生成的 `statementId`；
- `verified_fact`：映射到本次 facts review 中存在且状态为 `verified` 的 `claimIds`，设置 `directionConfirmed: true`；
- `author_inference`：满足事实映射，并提供正文中实际出现的 `inferenceMarker`；
- `opinion` / `non_claim`：`claimIds` 必须为空，禁止 `directionConfirmed` 和 `inferenceMarker`；
- 四类都写明具体核对理由。

数字、日期、排名、增长/下降、实体动作和事件断言不能再通过空 `claims` 或笼统的 `noVerifiableClaimsReason` 隐藏。若首次调用同时返回其他审校问题，也要一并修复；改动正文会产生新的陈述集合，旧 statementLedgerHash/coverage 随 revision/hash 失效，必须重新获取服务端 ledger。

## 6. 安装自动 QA 运行环境

交付验证不再只信任调用方填写的视口结果。`mediaops.delivery.verify` 会对确切的 `article.html` 运行固定版本 Nu、Playwright/Chromium 和 axe，并生成哈希绑定的报告、截图及 PDF。

本地至少准备：

- Bun 1.3.11（发布 CI 的固定版本）；
- Java 17 或更新版本；
- 锁文件中的依赖；
- Playwright 1.61.1 对应的捆绑 Chromium 149.0.7827.55。

```bash
bun install --frozen-lockfile
bunx playwright install chromium
bun run qa:release
```

自动 QA 检查 320/375/768/1440 像素、浅色/深色偏好、横向溢出、白底、axe 违规与 incomplete 结果、WCAG 文本间距压力、200% 字号，以及 A4/Letter 打印 PDF。HTML 在检查前后必须保持相同 SHA-256。

自动检查不能替代人工对内容、认知可读性与真实平台编辑器清洗效果的判断。因此 `delivery.verify` 仍要求与 renderer 不同的可信 principal 提交覆盖移动、桌面/平板、宽屏和打印的视觉确认。

## 7. 安装态与渠道验收

从真实插件安装目录运行：

```bash
bun run qa:installed
```

该命令只认证它自身所在的安装根、固定依赖和生成夹具；在源码目录运行成功不能替代安装态成功。对微信等平台，还要把 `wechat-richtext.html` 放入真实编辑器保存草稿并复核标题层级、段落、图片、白底和平台二次清洗。插件不会替用户执行最终发布点击。

## 8. 可恢复发布包

`mediaops.publish.package` 使用准备/提交两阶段状态。任何含 `DO-NOT-PUBLISH.commit-pending` 的目录都不是可发布包。若返回 `action_required`：

1. 保留目录、`operationId`、`packageId` 和标记，不手工删除或改写；
2. 按响应中的 `retry.tool`/`retry.args` 重试；处于 preparing 时必须由创建操作的同一 `issuer:principalId` 执行；
3. `recoveryMode=new` 表示首次提交，`resumed` 表示恢复准备态，`idempotent` 表示复验已提交终态；
4. `PACKAGE_FINALIZATION_PENDING` 表示数据库已提交但标记清理未完成，仍不可发布，原请求可安全重试；
5. `PACKAGE_ABORTED` 表示审批或内容绑定已变化。保留或隔离带标记目录作为审计证据，创建新 revision 和新 approval；该 operation 不得恢复。

审批绑定已提交 package 后是终态，不能再撤销。发布包中的 `package-manifest.json` 按正式 Schema 验证，包只复制已冻结字节，不重渲染。

## 9. 不应回退的兼容方式

- 不要重新接受调用方填写的 actor 名称作为身份。
- 不要把 `sourceTier`/`isPrimary` 重新开放为请求字段。
- 不要在缺少 Java、Chromium 或 axe 时用人工勾选覆盖自动 QA 失败。
- 不要把旧 JSONL 复制追加到 SQLite 导出结果。
- 不要把 `low_risk`、Nu/axe 通过或微信草稿保存分别描述为“原创证明”“完整 WCAG 认证”或“已发布”。
